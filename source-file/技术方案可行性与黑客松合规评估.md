# Life++ 技术方案本地落地与黑客松交付执行方案

> 执行口径：以 `source-file/技术方案.md` 为唯一基线。目标是完成“可演示、可测试、可验证”的黑客松本地开发与测试落地。

## 1. 总体结论

- 当前技术方案具备直接落地条件，下一步应进入工程实施，而不是继续讨论方案本身。
- 交付重点应聚焦 P0：**真实可跑通闭环 + 可验证证据 + 稳定演示脚本**。
- 验收标准应围绕三件事：
  1) 本地可启动；
  2) 关键链路可测试；
  3) 结果可被第三方复现与核验。

## 3. 可测试落地（P0 验收测试矩阵）

## 3.1 后端 API Smoke（必须自动化）

- `POST /auth/register`、`POST /auth/login`
- `POST /agents`、`GET /agents`
- `POST /agents/{id}/memories`、`POST /agents/{id}/memories/search`
- `POST /agents/{id}/chat`
- `POST /tasks`、`POST /tasks/{id}/accept`、`POST /tasks/{id}/complete`
- `GET /network/agents`、`GET /network/connections`

**通过标准**：
- 状态码正确；
- 关键字段完整；
- 任务状态流转符合预期（created → accepted → completed）。

## 3.2 前端关键流程（至少 1 条 e2e）

流程：
1. 登录并进入 Dashboard；
2. 创建 Agent；
3. 在 AgentChat 发送消息并获得回复；
4. 在 MemoryViewer 搜索到刚写入的记忆；
5. 在 Marketplace 发布并完成任务；
6. 在 NetworkGraph 可见 Agent 节点与关系变化。

**通过标准**：
- 页面无阻断错误；
- 主链路 1 次走通；
- 关键状态与后端数据一致。

## 4. 可验证落地（评审可核验证据）

## 4.1 证据清单（必须准备）

1. 启动证据：
   - 启动命令与成功日志；
   - 版本与环境信息（Python/Node）。

2. 测试证据：
   - smoke test 输出；
   - e2e 关键步骤截图/录屏。

3. 业务闭环证据：
   - Agent 创建结果；
   - 记忆写入与召回结果；
   - 任务全流程状态变化记录。

4. 链上/网络证据：
   - 与 Revive 交互结果（交易哈希、事件、状态）；
   - 如使用演示种子数据，必须显式标注“演示预置数据”。

## 4.2 文档交付（必须）

- `docs/demo-runbook.md`：3~5 分钟演示脚本（含失败回退方案）。
- `docs/test-report.md`：测试命令、结果、已知限制。
- `README.md`：从零到跑通的最短路径（10 分钟内可复现）。

---

## 5. 黑客松执行优先级（严格按 P0/P1）

## P0（必须完成）

1. 工程骨架 + 一键启动。
2. P0 接口与页面联调完成。
3. smoke test + 最小 e2e 打通。
4. 演示脚本、测试报告、证据材料齐备。

## P1（加分）

1. 多 Agent 并行/串行协作演示。
2. 流式响应与信誉可视化增强。
3. 异常降级（RPC 不稳定时的兜底策略）。

## 7. 最终验收标准（黑客松提交前）

满足以下条件即视为“可演示、可测试、可验证”：

- 一键启动成功，评委可本地复现；
- P0 主链路可连续演示成功；
- 自动化 smoke tests 全部通过；
- 关键流程有截图/日志/状态证据；
- 文档完整且与实际运行一致。
